# Step 1: Define the Problem

Getting a crystal-clear definition is the critical starting point. A surprising number of failures originate in poor problem definition — teams surge into data gathering without being clear about boundaries, criteria for success, time frame, or accuracy required.

## Characteristics of Good Problem Statements

- **Outcomes focused**: Express the problem in outcomes, not activities or intermediate outputs
- **Specific and measurable** wherever possible
- **Clearly time-bound**
- **Addresses decision-maker values and boundaries**, including accuracy needed and scale of aspirations
- **Allows sufficient scope for creativity** — too narrowly scoped problems artificially constrain solutions
- **Solved at the highest level possible** — for the organization as a whole, not just optimized for a part

The mnemonic **SMART** covers most of these: Specific, Measurable, Action-oriented, Relevant, Timely.

## Problem Definition Worksheet

Help the user fill out each element:

| Element | Description |
|---------|-------------|
| **Problem Statement** | Clear, outcomes-focused statement of what needs to be solved |
| **Context & Background** | What is the situation? What changed? |
| **Decision Maker** | Who will act on this? What are their values? |
| **Criteria for Success** | How will we know we've solved it? What's the threshold? |
| **Boundaries / Scope** | What's in and out of scope? What constraints exist? |
| **Time Frame** | When is the answer needed? Over what period does the solution apply? |
| **Accuracy Required** | Back-of-envelope? Directional? Precise? |
| **Key Stakeholders** | Who else is affected? Whose buy-in is needed? |

## Techniques for Sharpening Problem Statements

### Porpoising
Dive into some preliminary data, then surface to refine the problem statement. Repeat. Each iteration sharpens your understanding. Set up questions that must be answered to get the required specificity:
- Is this true for all segments or just some?
- Is this a supply or demand problem?
- Is the issue incidence or severity?
- What's the real scope — local, regional, global?

### Challenge with an Antithesis
Test robustness by posing the opposite: If the problem is "We need to expand," challenge with "What if we need to consolidate?" The antithesis might be about customer behavior, competitor reaction, or regulatory stance.

### Solve at the Highest Level
Often the presenting problem is a symptom of something bigger. A request to "evaluate our $6B capital plan" might really be about "inability to generate cash because of an uncompetitive cost position." Always ask: Is there a higher-level problem this is part of?

### Widen the Aperture
Allow flexibility in scope. Narrowly scoped problems yield fast answers but keep blinders on. Breakthrough ideas emerge from fresh perspectives and relaxed constraints.

### Integrate Design Thinking
When the user experience is at the core of the problem, integrate design thinking's empathize-define-ideate-build-test-deliver cycle. Spend time understanding the user's pain points through patient, sustained interaction.

## Iterating the Problem Statement

Problem statements should evolve as understanding deepens. Help the user:
1. Write a first-cut problem statement
2. Identify 3-5 questions that must be answered to sharpen it
3. Do quick research/interviews to answer those questions
4. Rewrite the problem statement
5. Assess quality against the criteria above

## Example: Coaching a User

When a user says "I'm trying to figure out whether to launch a new product":

1. Ask: "Who is the decision maker and what are their criteria for a good answer?"
2. Ask: "What's in scope? Just the launch decision, or also pricing, timing, channel?"
3. Ask: "What's the time frame for the decision and the accuracy you need?"
4. Ask: "What constraints exist — budget, team capacity, competitive timing?"
5. Draft a problem statement together and stress-test it
6. Ask: "Is there a higher-level problem — like whether the business model needs to change?"

## Red Flags in Problem Definitions
- No time frame specified
- No clear decision maker identified
- Problem is stated as an activity ("We need to do market research") rather than an outcome ("We need to know whether market X can support profitable entry")
- Scope is so narrow it precludes creative solutions
- Scope is so broad it's unactionable
